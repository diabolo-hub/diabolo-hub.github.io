git reset代码回滚

置顶 | 发表于 2023-05-04 | 分类于 git ， 笔记
字数统计: 3.2k | 阅读时长 ≈ 3 mins.

本地仓库分支切换到需要回退到代码分支，如： master

git checkout master

查找分析要回退到版本及hash码

怎么才能看懂 git log的线？

  1. 查看当前分支到log线
    git log –oneline –graph –decorate
  2. 查看两个分支间的关系线
    git log –oneline –graph –decorate master dev
  3. 查看指定分支hash码(b4ad2d0)的父节点等信息
    git log -1 –pretty=raw b4ad2d0

版本文件回退

  1. 在ideal中，git日志控制台中，选中需要回退到的版本记录位置，
    右键：将当前分支重置到此处…
    然后弹窗选择：硬 （状态回退到此处，且文件也变更回退到此处），本地有文件未提交的会丢失
    然后点击确定，本地仓库文件即已经回退

  2. hash码回退
    右键项目->git->重置HEAD（reset head）-> 将提交的位置输入要回退到的版本的hash码
    此处reset_type:选择 Hard (硬，与上面理由一样）
    验证可以查看具体回退的内容，确定（reset）即可

  3. 命令回退
    git reset –hard b25b0411b9d00e7aafd7b164b499be293a1bb942
    git reset –soft（软） HEAD^

  4. git 利用tag回退版本及合并两个远程分支
    git 利用tag回退版本及合并两个远程分支
    git回退到tag
    git tag使用

如何推送

git push -f
你可以简单地用–force（或-f）（*）解决这个问题：
git push –force
您可能需要再次设置上游：
git push –force –set-upstream origin
例如： git push –force –set-upstream origin master

git reset后无法推送–soft HEAD ^

注意点

  1. git push –set-upstream origin master
    如果有分支保护，不能–force
  2. git push –force –set-upstream origin master
    必须取消分支保护
  3. 需要 –set-upstream 才能推到远程，否则一直推不上去，需要将远程代码合并merge下来（远程还是未合并的代码，又会重复覆盖本地），一直循环
    git push –force –set-upstream origin master

k8s pod启动命令配置和dockerfile启动命令配置执行顺序

字数统计: 1.1k | 阅读时长 ≈ 1 mins.

k8s pod启动命令配置和dockerfile启动命令配置执行顺序

要知道，linux的指令是分为命令 和参数的，比如ls -l，ls 是命令，-l是参数，command是命令，args是参数，k8s的command和args其实是为了覆盖Fockerfile的ENTERPOINT功能

  • 如果command和args都没写，使用Dockerfile配置
  • 如果command写了， 那么Dockerfile的默认ENTRYPOINT配置会被忽略，执行command
  • 如果command没写，args写了， 那么Dockerfile的默认ENTRYPOINT配置会执行，使用k8s的args参数
  • 如果command和args都写了， 那么Dockerfile的默认ENTRYPOINT配置会被忽略，执行command并追加上args参数

git merge过程操作

置顶 | 发表于 2018-10-05 | 分类于 git ， 笔记
字数统计: 1.5k | 阅读时长 ≈ 1 mins.

背景

例如： master为上次更新代码，Dev为上次master代码更新之前的代码，且master更新之后dev也有代码改动。
现需要测试dev，将dev上线且合并到master。
首先需要将上次上线的master合并到当前dev中。

步骤

  1. 在gitLab中使用merge： master merge into dev

本地操作：

  1. 先将本地master更新到最新
    git checkout master
    git pull

  2. 将本地dev更新到最新
    git checkout dev
    git pull

  3. 合并
    git merge –no-ff master
    从 master 分支 合并到当前分支

–no-ff模式，普通合并，有历史，看得出做过合并
fast forward模式，快速合并，看不出做过合并
通常，合并分支时，Git尽可能用 fast forward 模式，但这种模式，删除分支后，会丢掉分支信息。

  1. 查看信息
    git status

  2. 处理差异
    git diff | 使用ideal 处理

  3. 提交
    git commit
    git push

git pull拉不下数据处理

git fetch origin

mysqlbinlog-日志管理工具

发表于 2023-01-04 | 分类于 database ， mysql ， notes
字数统计: 1.4k | 阅读时长 ≈ 1 mins.

语法

mysqlbinlog [options] log_file1 log_file2…

选项

–base64-output=name：决定binlog输出格式选项，
auto–自动显示binlog语句所有格式；
never–禁止显示binlog语句；
decode-rows–禁止显示binlog语句

–database=db_name, -d db_name：只显示指定数据库的日志

–verbose,-v：从行事件中重建伪 sql 语句

–verbose –verbose, -vv：从行事件中重建伪 sql 语句，添加对数据类型的注释

–start-datetime=datetime –stop-datetime=datetime：指定日期间隔的日志

–start-position=N –stop-position=N：指定事件位置间隔的日志

–skip-gtids[=(true|false)] ：不显示GTID

举例

  1. 创建新日志，对mysql和数据库做DML
    flush logs
  2. 不加参数显示binlog
    mysqlbinlog mysql-bin.000017
  3. 加参数显示的binlog日志的SQL执行记录
    mysqlbinlog mysql-bin.000017 -d test –base64-output=decode-rows –skip-gtids -vv | grep -B 1 -i ‘insert into’
