解决将已有git远程仓库的项目复制到另外一个已有远程连接的本地仓库，并提交的方法

发表于 2018-12-07 | 分类于 git ， 异常
字数统计: 1.5k | 阅读时长 ≈ 1 mins.

背景

项目A，已有远程仓库，可正常pull、push
项目B，已有远程仓库，可正常pull、push

操作

  1. 将项目B复制到项目A项目中，并git add . ， git commit

    结果

    在项目A中，项目B无法在项目中提交，整个git add 和 git commit过程无法发现项目B

分析

  1. 在项目A中的项目B文件夹下，右键git bash here，通过命令窗口，我们使用git remote -v
  • 发现此项目B文件夹下的远程仓库链接存在，且为项目B原本的远程仓库

解决

  1. 删除项目A中项目B文件夹对应的远程仓库（项目B文件夹里执行）
    git remote remove origin
  2. 删除项目A中项目B文件夹里的git版本控制信息（项目B文件夹里执行）
    find . -name “.git” | xargs rm -rf

    此时git add . 会提示： fatal: in unpopulated submodule ‘B’

  3. 清除项目A中项目B文件夹的git缓存信息（此处需要在项目A文件夹中执行，这样才能找到项目B文件夹）
    git rm -rf –cached hexo-theme-matery

merge和rebase的区别

发表于 2018-08-19 | 分类于 git ， 笔记
字数统计: 949 | 阅读时长 ≈ 1 mins.
  1. （一股脑）使用merge命令合并分支，解决完冲突，执行git add .和git commit -m’fix conflict’。这个时候会产生一个commit。
  2. （交互式）使用rebase命令合并分支，解决完冲突，执行git add .和git rebase –continue，不会产生额外的commit。这样的好处是，‘干净’，分支上不会有无意义的解决分支的commit；坏处，如果合并的分支中存在多个commit，需要重复处理多次冲突。
  3. git pull和git pull –rebase区别：git pull做了两个操作分别是‘获取’和合并。所以加了rebase就是以rebase的方式进行合并分支，默认为merge。

git pull = git fetch + git merge
git pull –rebase = git fetch + git rebase
